# Аудит конфликтующих и повторяющихся логик

Дата: 2026-03-05

## Что проверено

- Поиск модулей с похожими зонами ответственности (`spawn*`, `build*`, `overlay*`, `pathfinding*`).
- Сопоставление реально используемых импортов в `src/scenes/Start.js` с альтернативными реализациями в репозитории.
- Ручной просмотр ключевых модулей на предмет дублирования алгоритмов и расхождений в поведении.

## 1) Спавн: две параллельные системы (`spawnManager` и `spawnSystem`)

### Наблюдение

- В рантайме сцена использует `spawnManager` (`findSpawn`, `loadSpawnRegistry`, `addSpawnToRegistry`, `clearSpawnRegistry`).
- В кодовой базе параллельно существует `spawnSystem` с почти теми же обязанностями: валидация тайла спавна, поиск безопасной позиции и реестр в `localStorage`.

### Почему это конфликт

- Оба модуля реализуют **один и тот же механизм** (поиск + реестр), но с разной API и разными правилами.
- Есть риск «тихого расхождения» логики: фиксы в одном модуле не попадают во второй.

### Расхождения, которые уже есть

- Ключ реестра:
  - `spawnManager`: глобальный `ANDREI_PROJ_SPAWNS_V1`.
  - `spawnSystem`: `spawns_${worldSeed}`.
- Стратегия поиска:
  - `spawnManager`: детерминированные попытки в диске + спираль до `R`.
  - `spawnSystem`: псевдослучайный квадрат с расширением диапазона + fallback-скан до большого радиуса.
- Ограничение дистанции:
  - `spawnManager`: внешний `safeDistTiles`.
  - `spawnSystem`: собственный clamp (`200..1000`) в `clampSpawnMinDist`.

### Рекомендация

Оставить **один** источник истины для спавна (предпочтительно `spawnManager`, т.к. он фактически используется), а `spawnSystem`:
1. либо удалить,
2. либо превратить в тонкий адаптер, проксирующий в `spawnManager`.

## 2) Строительство: `buildManager` (через sim) vs `buildSystem` (локальная логика)

### Наблюдение

- Сцена создает `BuildManager` и работает через симуляцию (`validatePlacement`, `placeBuilding`).
- В проекте есть отдельный `buildSystem` с собственной моделью зон, занятости, ограничений первого здания и проверок поверхности.

### Почему это конфликт

- Это две разные реализации одного процесса размещения построек.
- Правила валидации и зона строительства могут расходиться, если кто-то начнет переиспользовать `buildSystem` в UI/тестах.

### Критичное расхождение

- `BuildManager` фактически доверяет `sim` как единому источнику правил.
- `buildSystem` поддерживает отдельные поля (`buildings`, `zones`, `setSpawn`, `canPlace`) и собственные параметры (`house1BuildRadius`, `buildMaxSlope`) — потенциально не совпадающие с балансом из `sim`.

### Рекомендация

Зафиксировать архитектурно, что **единственный** pipeline строительства — через `sim`, а `buildSystem`:
1. либо удалить как legacy,
2. либо перенести в explicit `legacy/` и пометить как неиспользуемый.

## 3) Оверлеи: две реализации `OverlayManager` в разных директориях

### Наблюдение

- Используется `src/world/game/overlay/overlayManager.js` (чанк-рендеринг текстур).
- Параллельно лежит `src/world/game/ui/overlayManager.js` (graphics-рендер district/influence/routes).
- Оба экспортируют класс с одинаковым именем `OverlayManager`.

### Почему это конфликт

- Это функционально пересекающиеся менеджеры визуальных оверлеев с одинаковым именем.
- Легко импортировать «не тот» модуль по ошибке.
- Расходятся интерфейсы (`setFlags`/`setRouteSelection` vs `setToggles`/`setPlacementType`).

### Рекомендация

- Убрать двусмысленность имен:
  - например, `LegacyOverlayManager` для старой реализации.
- Закрепить один публичный entrypoint для оверлеев.
- Если UI-версия не используется — удалить.

## 4) Патфайндинг: дублирующиеся A* ядра для юнитов и торговых маршрутов

### Наблюдение

- `ui/pathfinding.js` и `sim/routePathfinding.js` содержат очень похожие блоки:
  - Manhattan heuristic,
  - open set / gScore / parent,
  - line fallback,
  - ограничения по `maxNodes`, `margin`, `maxDistance`.

### Почему это важно

- Это не прямой конфликт, но **повторяющийся механизм** высокого риска: багфикс в A* нужно синхронно переносить в два места.

### Рекомендация

Вынести общее ядро в модуль наподобие `sim/shared/aStar.js` с инъекцией `passableFn` и параметров, а в текущих файлах оставить domain-specific обертки.

---

## Приоритет исправлений

1. **Высокий**: убрать дубли `spawn*` и `overlay*` (наиболее вероятные источники путаницы и регрессий).
2. **Средний**: определить судьбу `buildSystem` (legacy vs удаление).
3. **Средний**: унифицировать A* ядро (снижение техдолга).

## Быстрые безопасные шаги (без изменения геймплея)

1. Пометить неиспользуемые модули префиксом `legacy-` + комментарий в заголовке.
2. Добавить lint-правило/скрипт на поиск «неиспользуемых экспортируемых модулей». 
3. Создать один `index`-entrypoint для spawn/build/overlay API, чтобы исключить случайный импорт альтернативной реализации.
